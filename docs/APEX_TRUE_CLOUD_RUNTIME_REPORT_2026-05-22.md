# APEX True Cloud Runtime Report - 2026-05-22

## Scope

MVP only. The white-label project was not deployed or modified.

## Target

- APEX host: `https://apex.oraclecorp.com/pls/apex`
- Workspace: `EMEAWJ`
- Application ID: `56174`
- Application alias: `nsdemohelper`
- Runtime URL: `https://apex.oraclecorp.com/pls/apex/r/emeawj/nsdemohelper`

## Root Cause

The cloud application was imported into official Oracle APEX, but the deployed runtime was not the real MVP UI. Earlier packages either redirected to the local MVP runtime or used a simplified cloud shell that did not include the full local MVP markup, CSS, JavaScript behavior, and static assets.

That meant APEX was not yet acting as the true UI host. The working MVP experience still lived in the local Node runtime, while the cloud version was visually incomplete and did not route API calls consistently to the local Codex runtime.

## Localhost Dependency Findings

Frontend/runtime dependencies found before the fix:

- earlier `apex/apps/f56174/pages/p00001-home.apx` builds redirected to `http://localhost:4173`
- the full MVP UI was generated by the local Node server in `src/control-panel.mjs`
- browser UI actions in the full local MVP call local `/api/...` endpoints
- the ribbon/static asset was previously served by the local Node server from `/assets/...`
- the cloud browser needed CORS and private-network permission headers before `https://apex.oraclecorp.com` could call the local bridge reliably

Intentional local-only areas that remain outside the cloud frontend:

- optional Codex/local provider calls
- local NetSuite browser automation
- local voice/demo runner behavior
- local full Node MVP for desktop usage

## Fix Applied

The cloud APEX package now includes:

- `apex/apps/f56174/shared-components/static-files/nsdemohelper-cloud.html`
- `apex/apps/f56174/shared-components/static-files/b31e8773d38eab8a3c4a3e25413a77595a765691.png`
- `scripts/build-apex-cloud-runtime.mjs`
- Page 1 redirect to the APEX-hosted static application file, not localhost:

```plsql
apex_util.redirect_url('/pls/apex/' || :APP_IMAGES || 'nsdemohelper-cloud.html');
```

The static cloud page is generated from the working local MVP HTML so the cloud experience keeps the same NetSuite-style layout, tabs, forms, admin UI, intelligence pages, and generated-output pages as the local MVP.

The cloud build injects a small bridge:

- frontend routing and static assets stay on `apex.oraclecorp.com`
- browser calls to relative `/api/...` endpoints are redirected to the configured local bridge, defaulting to `http://localhost:4173`
- export/download links are routed through the same bridge when needed
- the ribbon image is hosted as an APEX static application file

The local MVP server now allows APEX-origin API calls with CORS/private-network headers for `https://apex.oraclecorp.com`.

## Remaining Local Components

The full Codex-backed local MVP remains available through `npm run mvp`.

In the cloud runtime, localhost is used only for API/Codex bridge calls. It is not used for frontend hosting, routing, or static assets.

## Remaining Limitations

The APEX cloud page uses the current user's browser to call `http://localhost:4173`. That works for this machine and for colleagues who also run the local MVP/Codex bridge on their own machine. For a fully shared internal application where colleagues do not run anything locally, the next step is a secure hosted provider bridge or an Oracle-managed backend endpoint.

For full cloud parity and easier colleague sharing, future work should add:

- a secure hosted or connector-based Codex provider
- APEX/ORDS-backed session persistence if logs should live in cloud instead of the local MVP database
- protected cloud admin configuration backed by APEX database objects
- a setup check that tells each user whether their local bridge is reachable
